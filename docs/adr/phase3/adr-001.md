# ADR-001: Visualization Session API MVP for Phase 3

## Status

Accepted

## Date

2026-06-27

## Context

Phase 3 must expose Ymir's `World` simulation state to external clients. The roadmap
identifies WebSocket + Protobuf, scenario loading, basic control commands, and real-time
state transmission as the intended control API. Codebase research shows that Phase 2
already provides the required simulation foundation through `World::step(dt)`,
`World::snapshot()`, `Environment`, `NavalDomain`, and `CouplingRegistry`, while
`apps/server` remains an empty application target.

The selected primary user is a visualization client. The MVP must provide body
kinematics, vessel operational state, and environment state. The client needs basic
session control: load scenario, start, pause, resume, stop, and observe state. The
preferred stream behavior is event-first updates rather than periodic full snapshots,
and the main success criterion is integration completeness.

## Decision

Phase 3 will use a Visualization Session API MVP. The product will let a visualization
client load a scenario, control the simulation session lifecycle, receive an initial
full session snapshot, and then receive event-first updates covering body kinematics,
vessel operational state, and environment state.

The MVP will avoid live scenario editing and detailed operator controls. It will
prioritize one complete end-to-end integration path over breadth.

## Alternatives Considered

### Alternative 1: Control-Rich Operator API

- **Description**: Include session control plus live changes to environment, waypoints,
  vessel modes, rudder commands, and thruster commands.
- **Pros**: More useful for training operators and interactive maneuver experiments.
- **Cons**: Expands scope into operational control semantics before the visualization
  contract is stable.
- **Why rejected**: It risks turning Phase 3 into a broad command product instead of
  the smallest complete external integration layer.

### Alternative 2: Stream-Only Viewer API

- **Description**: Let external code create and run simulations while the server only
  streams state/events to visualization clients.
- **Pros**: Fastest path to rendering live state.
- **Cons**: Does not let the client load or control a full World session.
- **Why rejected**: It fails the selected success criterion of integration completeness.

## Consequences

### Positive

- Delivers a complete external client workflow with limited product scope.
- Gives visualization clients enough state to render bodies, vessel status, and environment.
- Establishes the message contract future operator controls can extend.
- Keeps Phase 3 aligned with the existing Phase 2 `World` snapshot model.

### Negative

- Does not yet support live operator editing of environment, waypoints, or actuator commands.
- Visualization clients may still need client-side interpolation when event cadence is lower
  than render cadence.
- Event-first behavior requires clear baseline snapshot semantics so clients can recover
  from missed or out-of-order updates.

### Risks

- **Risk**: Stakeholders may expect direct vessel control in Phase 3.
  **Mitigation**: Document live scenario and vessel command editing as post-MVP scope.
- **Risk**: Event-first updates may be insufficient for early visualizers expecting
  fixed-rate snapshots.
  **Mitigation**: Require an initial full snapshot and allow clients to request current
  state on demand.
