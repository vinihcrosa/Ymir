# ADR-002: std::variant para Despacho de Controlador em DynamicVessel

## Status

Aceito

## Data

2026-06-27

## Contexto

`DynamicVessel` precisa suportar três controladores mutuamente exclusivos:
`ManeuverController`, `PrescribedController`, e `BerthManeuverSystem`. A cada tick, o
controlador ativo é invocado via `updateControl()`. É necessária uma decisão sobre o
mecanismo de seleção e despacho.

Os três controladores têm assinatura de update idêntica:
```cpp
void update(double t, double dt, const NavalContext& ctx,
            std::vector<Thruster>& thrusters, std::vector<Rudder>& rudders);
```

Isso satisfaz o requisito para duck typing via `std::visit`.

## Decisão

`VesselController` é definido como:
```cpp
using VesselController = std::variant<
    ManeuverController,
    PrescribedController,
    BerthManeuverSystem
>;
```

O despacho em `DynamicVessel::updateControl()` usa `std::visit` com lambda genérico:
```cpp
std::visit([&](auto& ctrl) {
    ctrl.update(t, dt, ctx, thrusters_, rudders_);
}, controller_);
```

Nenhuma classe base virtual. Nenhuma interface `IController`. Os três tipos satisfazem
um conceito implícito de `VesselController` (método `update` com a assinatura correta).

## Alternativas Consideradas

### Alternativa A: Classe base virtual IController

- **Descrição**: Herança polimórfica tradicional com `virtual void update(...)` e
  `std::unique_ptr<IController>` em DynamicVessel.
- **Prós**: Extensível fora do codebase (qualquer classe pode implementar IController);
  sem recompilação ao adicionar novo controlador.
- **Contras**: Alocação heap por troca de controlador; vtable overhead em hot path (tick);
  controladores são exclusivos do domínio Ymir — extensibilidade externa não é requisito.
- **Por que rejeitada**: Overhead sem benefício. Os três controladores são tipos fechados
  conhecidos em tempo de compilação. Adicionar um quarto controlador requer modificação
  controlada de qualquer forma.

### Alternativa B: std::function + enum ControlMode

- **Descrição**: DynamicVessel armazena enum `ControlMode` e `std::function<void(...)>`
  para o update atual.
- **Prós**: Mais simples de entender à primeira vista.
- **Contras**: Perde type safety; estado interno do controlador precisa de mecanismo
  separado (captures ou struct externa); `std::function` tem custo de type erasure.
- **Por que rejeitada**: std::variant preserva o tipo concreto, permitindo acesso ao estado
  interno do controlador (e.g., `BerthManeuverSystem::currentPhase()`) sem cast extra.

## Consequências

### Positivas

- Zero alocação heap no tick de controle.
- State do controlador ativo é acessível diretamente via `std::get<BerthManeuverSystem>(controller_)`.
- Erro de compilação imediato se novo controlador não implementar a assinatura correta.
- Troca de controlador em runtime: `vessel.setController(PrescribedController{cfg})`.

### Negativas

- Adicionar um quarto controlador exige alterar o alias `VesselController` e recompilar
  tudo que inclui `DynamicVessel.h`.
- `std::visit` tem custo de branch (switch sobre o índice do variant) — aceitável
  dado que é chamado 1× por tick, não no loop interno.

### Riscos

- Risco: BerthManeuverSystem internamente usa ManeuverController para a fase Navigating —
  isso é composição interna, não dois controladores simultâneos. Não viola o modelo.
  Mitigação: documentar explicitamente que `VesselController` é sobre o controlador raiz,
  não subsistemas internos.
