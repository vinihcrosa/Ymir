# ADR-003: Thruster e Rudder como Entidades com Estado de Atuador

## Status

Aceito

## Data

2026-06-27

## Contexto

Em `ThrustForces` (libs/physics/) e `RudderForces` (libs/physics/), o estado dinâmico
de atuadores (RPM atual, ângulo atual) é gerenciado internamente pelas próprias classes
de força. `ThrustForces` mantém `currentRPM_` (array de doubles) e aplica o filtro de
primeira ordem em `computeNaval()`. `RudderForces` mantém `currentAngle_` e aplica o
rate limiter em `computeNaval()`.

Dois problemas arquiteturais:

1. **Estado de atuador não é testável em isolamento**: para verificar que o filtro de
   RPM produz o valor correto após 1s, o teste precisa instanciar ThrustForces completo
   (com VesselConfig, rho, etc.) e rodar `computeNaval()` — que também aplica forças no
   corpo. Não há como testar a dinâmica do atuador separadamente.

2. **Mutação de config em runtime**: controladores escrevem a demanda em
   `ThrusterConfig.commandRPM` — campo de configuração sendo usado como variável de
   estado. Config deve ser imutável após construção.

## Decisão

Criar entidades `Thruster` e `Rudder` em `libs/vessel/`. Cada entidade possui:
- Referência à config estática (`ThrusterConfig` ou `RudderConfig`): parâmetros físicos imutáveis.
- Estado dinâmico separado (`ActuatorState`): valores atuais e demandados, mutáveis via `update(dt)` e `setDemand()`.
- Método `update(double dt)`: toda a lógica de dinâmica do atuador (filtro de 1ª ordem,
  rate limiter) vive aqui — não em `computeNaval()`.

`ThrustForces` e `RudderForces` em libs/physics/ são refatorados para:
- Remover estado interno (`currentRPM_`, `currentAngle_`).
- Adicionar método `setActuatorState(id, cmd)` que recebe o estado atual do atuador.
- `computeNaval()` lê de `commands_[i]` (estado externo) em vez de computar o filtro.

`DynamicVessel::syncToForceModels()` é responsável por transferir o estado das entidades
para os modelos de força antes do passo de integração.

Para evitar dependência circular (libs/physics/ não pode incluir libs/vessel/), os tipos
de comando são definidos dentro das próprias classes de força em libs/physics/:

```cpp
// Em ThrustForces.h
struct ThrusterCommand {
    double currentRPM         = 0.0;
    double currentAzimuth_deg = 0.0;
    double currentPitch       = 0.0;
};
void setActuatorState(std::size_t id, const ThrusterCommand& cmd);
```

```cpp
// Em RudderForces.h
struct RudderCommand {
    double currentAngle_rad = 0.0;
};
void setActuatorState(std::size_t id, const RudderCommand& cmd);
```

`Thruster::syncCommand()` converte o `ActuatorState` interno para `ThrusterCommand` —
tipo estável que libs/vessel/ importa de libs/physics/.

## Alternativas Consideradas

### Alternativa A: Manter estado em ThrustForces/RudderForces, adicionar getters

- **Descrição**: Adicionar `currentRPM(id)`, `currentAngle(id)` aos modelos de força.
  `DynamicVessel` lê de volta para expor estado.
- **Prós**: Mudança mínima. Sem novos tipos.
- **Contras**: Estado continua em libs/physics/, não testável em isolamento. A pergunta
  "qual RPM está o propulsor agora?" requer instanciar um modelo de força completo.
- **Por que rejeitada**: Não resolve o problema de testabilidade nem a mutação de config.

### Alternativa B: Mover ThrustForces e RudderForces para libs/vessel/

- **Descrição**: Transferir as duas classes para libs/vessel/ onde elas podem referenciar
  Thruster e Rudder diretamente.
- **Prós**: Sem necessidade de tipos intermediários; acoplamento direto e limpo.
- **Contras**: Quebra os caminhos de include de todos os consumidores existentes;
  move responsabilidade de física para o layer de vessel.
- **Por que rejeitada**: Custo de migração desproporcionalmente alto para uma questão de
  testabilidade que é resolvida com menos invasividade pela abordagem eleita.

### Alternativa C: ThrusterState/RudderState em libs/common/

- **Descrição**: Definir os structs de estado em libs/common/ onde ambos os layers podem importar.
- **Prós**: Sem tipos duplicados; sem dep circular.
- **Contras**: libs/common/ acumula tipos de domínio que não são genéricos (ThrusterState
  é claramente naval); polui o namespace de utilitários.
- **Por que rejeitada**: Violaria a responsabilidade de libs/common/ (tipos matemáticos e
  constantes físicas, não estado de domínio).

## Consequências

### Positivas

- Dinâmica de atuador testável em isolamento: `Thruster t(cfg); t.setDemand(100); t.update(1.0); REQUIRE(t.state().currentRPM == Approx(...))`.
- `ThrusterConfig` e `RudderConfig` são imutáveis após construção do objeto.
- `computeNaval()` em ThrustForces/RudderForces torna-se função pura (dado o estado externo) — mais previsível e cacheable.
- DRY: a lógica do filtro de 1ª ordem existe em exatamente um lugar (Thruster::update).

### Negativas

- Refatoração de ThrustForces/RudderForces quebra os testes existentes que dependem do
  comportamento de update interno a `computeNaval()`.
- `DynamicVessel::syncToForceModels()` cria acoplamento estrutural: DynamicVessel precisa
  conhecer ThrustForces e RudderForces (ponteiros não-owners).

### Riscos

- Risco: `syncToForceModels()` chamado fora de ordem (antes de updateStates()) produz
  forças incorretas. Mitigação: NavalSimulation sempre chama na sequência
  `updateControl → updateStates → syncToForceModels → sim_.step()`.

## Notas de Implementação

`VesselConfig.ThrusterConfig.commandRPM` é removido e substituído por `initialRPM`
(estado inicial do atuador, padrão = 0.0). O construtor de `Thruster` usa `initialRPM`
para inicializar `state_.currentRPM`.
