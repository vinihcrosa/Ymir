# ADR-003: One CMake Static Library Target Per Bounded Context

## Status

Accepted

## Date

2026-06-25

## Context

The current build produces two static libraries: `ymir_core` and `ymir_naval`. After splitting into six bounded context directories, a decision is needed on how many CMake targets to produce.

Options range from keeping two monolithic targets to creating one target per context (six targets).

## Decision

Create one CMake `STATIC` library target per bounded context:

| Target | Directory | Links to |
|--------|-----------|----------|
| `ymir_common` | `libs/common/` | (none â€” no internal deps) |
| `ymir_physics` | `libs/physics/` | `ymir_common`, `SUNDIALS::cvode` |
| `ymir_simulation` | `libs/simulation/` | `ymir_physics`, `ymir_common` |
| `ymir_world` | `libs/world/` | `ymir_physics`, `ymir_common` |
| `ymir_vessel` | `libs/vessel/` | `ymir_physics`, `ymir_common` |
| `ymir_persistence` | `libs/persistence/` | `ymir_physics`, `ymir_world`, `ymir_vessel`, `nlohmann_json::nlohmann_json` |

Each target declares its public headers via `target_include_directories(... PUBLIC ...)`. Consumers link to specific targets rather than a monolithic library.

## Alternatives Considered

### Alternative 1: Keep Two Targets (ymir_core + ymir_naval)

- **Description**: Rename directories but keep `ymir_core` (libs/core/) and `ymir_naval` (libs/naval/) as the two CMake targets
- **Pros**: Minimal CMake changes; no dependency graph changes; low risk
- **Cons**: Target names no longer reflect bounded context names; the split is visible only in directories, not enforced by the linker; a test for a world module can still accidentally link to vessel types because both are in `ymir_naval`
- **Why rejected**: Cross-context coupling is only detectable at runtime, not at link time; defeats the architectural intent of the split

### Alternative 2: Single Monolithic Target

- **Description**: One `ymir` target containing all code
- **Pros**: Simplest build
- **Cons**: No boundary enforcement whatsoever; does not reflect the architecture
- **Why rejected**: Trivially ruled out

## Consequences

### Positive

- The linker enforces module boundaries: a target that doesn't link `ymir_world` cannot accidentally use wave engine types
- Test executables link only what they test (faster build, clearer scope)
- Future modules can be added as new targets without modifying existing ones

### Negative

- Six `CMakeLists.txt` files instead of two â€” more maintenance
- Must carefully specify `PUBLIC` vs `PRIVATE` link dependencies to avoid over-exposure
- `ymir_persistence` must link everything it depends on â€” if the dependency graph grows, this becomes verbose

### Risks

- **Circular dependency**: `ymir_world` uses physics types; `ymir_physics` must NOT use world types. Enforce this via the dependency graph (world depends on physics, not vice versa). CMake will error on circular deps.

## Implementation Notes

Use CMake alias targets (`add_library(Ymir::Physics ALIAS ymir_physics)`) so consumers can write `target_link_libraries(foo Ymir::Physics)` â€” this is the pattern used by LLVM and Boost.
