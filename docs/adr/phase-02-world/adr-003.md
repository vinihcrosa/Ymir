# ADR-003: IDomain Interface em libs/world para Evitar Dependência Circular

## Status

Accepted

## Date

2026-06-27

## Context

`World` (in `libs/world/`) must store a heterogeneous list of domains polymorphically.
`NavalDomain` (in `libs/simulation/`) must implement that interface. If `IDomain` lives
in `libs/simulation/`, then `libs/world/` depends on `libs/simulation/`, and
`libs/simulation/` already transitively depends on `libs/world/` headers (e.g., `NavalEnvironment`).
This would create a circular CMake dependency.

The dependency graph must remain a DAG.

## Decision

`IDomain` is defined in `libs/world/include/ymir/world/IDomain.h`. It contains only four
pure virtuals: `onAddedToWorld(Environment&)`, `step(double dt)`,
`allBodyPositions() const`, and `bodyState(int id) const`, plus a virtual destructor.
No naval-specific types appear in this header.

`NavalDomain` in `libs/simulation/` includes `IDomain.h` from world and implements
the interface. `libs/simulation/CMakeLists.txt` adds `ymir_world` as a link target (was
previously an informal include-dir-only dependency). `libs/world/` does not depend on
`libs/simulation/`.

Final DAG after Phase 2:
```
ymir_common
  ↑
ymir_physics ← ymir_common
  ↑
ymir_world ← ymir_physics, ymir_common
  ↑
ymir_simulation ← ymir_physics, ymir_vessel, ymir_world, ymir_common
```

## Alternatives Considered

### Alternative A: IDomain in libs/common

- **Description**: Place IDomain in the shared `libs/common/` layer.
- **Pros**: No new dependencies.
- **Cons**: `libs/common/` is for math utils and physical constants — domain orchestration has no business there. Pollutes the common layer with simulation concerns.
- **Why rejected**: Violates module responsibility boundaries.

### Alternative B: World in libs/simulation

- **Description**: Move `World` to `libs/simulation/`, where IDomain and NavalDomain also live.
- **Pros**: All orchestration in one library.
- **Cons**: `libs/world/` already has wave forces, environment, and terrain-related types. Moving World to simulation splits related world-state concepts across two libraries. Also means the public API entry point (`World`) is buried inside the simulation library.
- **Why rejected**: World is conceptually in the world layer; NavalDomain is the naval implementation. Keeping them separate respects the existing layer split.

## Consequences

### Positive

- Dependency graph remains a strict DAG with no circular deps.
- Adding a future non-naval domain (e.g., `ElectricalDomain`) only requires implementing IDomain — no changes to World or libs/world.
- `libs/world/CMakeLists.txt` stays lean.

### Negative

- `libs/simulation/` now formally depends on `libs/world/` (currently informal via include dirs). This makes simulation slightly heavier to link independently.
