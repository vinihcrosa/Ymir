# ADR-001: World como Thin Orchestrator com DomainRegistry e NavalDomain como Primeiro Domínio

## Status

Accepted

## Date

2026-06-27

## Context

Phase 2 must promote the simulator from a single-vessel NavalSimulation to a multi-body world
where ships and tugs coexist with shared environment and inter-body force coupling.

Three structural options were evaluated for how the new `World` class should be positioned:
(A) thin orchestrator owning domains, (B) monolithic absorber of NavalSimulation, or (C) a
facade over an extended NavalSimulation. The decision determines extensibility for Phases 3–4
and whether future non-naval domains (structural, electrical) can be added without rewrites.

NavalSimulation currently owns environment, bodies, and the naval tick — all in one class.
This violates the separation needed for multi-domain orchestration.

## Decision

Introduce `World` as a thin orchestrator that owns:
- `GlobalClock` — single authoritative time source
- `Environment` — promoted from NavalSimulation, shared across all domains
- `DomainRegistry` — list of domains, each implementing a common `IDomain` interface
- `CouplingRegistry` — resolves inter-body force exchange after all domains step

`NavalDomain` is introduced in `libs/simulation/` as the first (and only) domain in Phase 2.
It replaces `NavalSimulation` as the naval body manager, managing N bodies each with a
`RigidBody6DOF`, `NavalContext`, and `DynamicVessel`.

`World::step(dt)` sequence:
1. Advance GlobalClock
2. Call `domain.step(dt)` for each registered domain
3. Resolve all CouplingPorts (Jacobi: forces from tick t applied at t+dt)
4. Flush resolved forces into each domain for next tick

`NavalSimulation` is kept as a thin wrapper around a single-body `NavalDomain` for one phase,
then deprecated in Phase 3.

## Alternatives Considered

### Alternative B: World Monolítico que Absorve NavalSimulation

- **Description**: World merges NavalSimulation's body management and tick logic. No NavalDomain concept.
- **Pros**: Fewer classes; simpler initial migration.
- **Cons**: World becomes a naval-specific god object. Non-naval domains cannot be added without modifying World. CouplingRegistry has no clean home.
- **Why rejected**: Violates single responsibility; blocks future domain extensibility that the roadmap explicitly plans.

### Alternative C: NavalSimulation Estendida + Coupling Layer

- **Description**: NavalSimulation kept; CouplingRegistry added externally; World is a thin facade.
- **Pros**: Smallest initial breaking change.
- **Cons**: NavalDomain never gets created. The roadmap's explicit substitution ("NavalDomain substitui NavalSimulation") never happens. Accumulates structural debt across phases.
- **Why rejected**: Defers the refactor to Phase 3 where it competes with the API work; the roadmap anticipates this substitution happening in Phase 2.

## Consequences

### Positive

- `World` has no naval-specific code — any future domain (non-naval) can be registered without modifying World.
- `NavalDomain` owns naval semantics; naval changes don't leak into the orchestration layer.
- `Environment` is shared once — no risk of divergent environment state across bodies.
- `CouplingRegistry` has a natural home at World level, mediating cross-domain interactions.

### Negative

- Two classes (`World` + `NavalDomain`) replace one (`NavalSimulation`): more files to maintain.
- `NavalSimulation` wrapper adds a short-lived compatibility layer (removed in Phase 3).

### Risks

- Risk: `World` grows into a god object if domains start calling back into it for physics. Mitigation: domains receive Environment by reference at construction; they must not hold a pointer to World.

## Implementation Notes

- `IDomain` interface in `libs/simulation/`: `step(double dt)`, `allBodyPositions()`, `bodyState(int id)`.
- `CouplingPort` holds: `Force6D force`, `Point3D interfacePosition`, `Velocity3D interfaceVelocity`, producer/consumer ids.
- Jacobi splitting is acceptable for naval dt (0.05–1.0s) where coupling dynamics are much slower than the integration step.
- `NavalDomain` header goes in `libs/simulation/include/ymir/simulation/NavalDomain.h`.
