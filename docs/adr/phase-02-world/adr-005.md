# ADR-005: Environment como Classe com Setters Tipados em libs/world; NavalEnvironment Deprecated

## Status

Accepted

## Date

2026-06-27

## Context

`NavalEnvironment` is currently a plain struct with public fields (no validation, no
encapsulation) owned by `NavalSimulation`. Phase 2 promotes environment to `World` level
where it must be shared across domains and mutated via a typed API (setters that validate
ranges and can trigger future events).

The question is whether to: (A) add setters to `NavalEnvironment` in place, (B) create
a new `Environment` class and deprecate `NavalEnvironment`, or (C) keep `NavalEnvironment`
as-is for backward compat and wrap it in a new adapter.

## Decision

Introduce `Environment` in `libs/world/include/ymir/world/Environment.h`. It holds the
same fields as `NavalEnvironment` but exposes only typed setters and const getters â€”
no public fields. `NavalEnvironment` is `[[deprecated]]` and becomes a type alias:
`using NavalEnvironment = Environment;`

`NavalDomain::buildContext()` is updated to read from `const Environment&` directly
(replacing the local `NavalEnvironment env_` member). `NavalDomain` receives
`Environment&` via `onAddedToWorld(Environment&)` (IDomain interface).

This eliminates the need for any bridge/conversion method and avoids data duplication.

## Alternatives Considered

### Alternative A: Add Setters to NavalEnvironment In-Place

- **Description**: Keep `NavalEnvironment` as-is, add setters, promote to `libs/world/`.
- **Pros**: Minimal change; no deprecation churn.
- **Cons**: `NavalEnvironment` name implies naval-specificity; as a World-level concept it
  should be domain-agnostic. Public fields remain (anyone can bypass setters).
- **Why rejected**: Naming and encapsulation gap; hard to add range validation or future
  event publishing without API breakage.

### Alternative C: Adapter Wrapper

- **Description**: `Environment` wraps `NavalEnvironment` with setters; `toNavalEnvironment()` bridges.
- **Pros**: Zero change to NavalEnvironment; backward compat without deprecation.
- **Cons**: Introduces data duplication (two structs with same fields); sync bugs inevitable.
- **Why rejected**: Unnecessary indirection and data duplication for a simple value type.

## Consequences

### Positive

- `Environment` is domain-agnostic â€” future non-naval domains (e.g., weather simulation)
  can share the same World environment.
- Setters can validate ranges (speed â‰Ą 0, depth > 0) without changing callers.
- Type alias `using NavalEnvironment = Environment` allows existing callers to compile
  without changes during the Phase 2 transition.

### Negative

- `NavalDomain::buildContext()` must be updated (find-replace `env_.field` â†’ `env_->field()`).
