# ADR-004: Refatoração de NavalSimulation para Suporte a N Corpos

## Status

Aceito

## Data

2026-06-27

## Contexto

`NavalSimulation` é construído com um único `std::unique_ptr<RigidBody6DOF>` e mantém
um ponteiro não-owner `body_ptr_` para esse corpo. O contexto naval (`NavalContext`),
o vetor de modelos de força (`navalModels_`) e a média EMA (`q_avg_`) são escalares —
associados a um único corpo.

Isso impede simular múltiplos corpos navais (navio + rebocadores independentes). A classe
`Simulation` interna já suporta N corpos via `addBody(id, unique_ptr<AbstractBody>)` e
`std::map<int, unique_ptr<AbstractBody>>`. O gargalo é exclusivamente `NavalSimulation`.

## Decisão

Refatorar `NavalSimulation` para suportar N corpos. O construtor de corpo único é removido
em favor de `addBody(id, unique_ptr<RigidBody6DOF>)`. O estado por corpo é agrupado em
uma struct interna `BodyEntry`:

```cpp
struct BodyEntry {
    RigidBody6DOF*                body   = nullptr; // não-owner (owned by sim_)
    std::vector<NavalForceModel*> models;
    NavalContext                  ctx;
    Vector6                       q_avg{};
    DynamicVessel*                vessel = nullptr; // opcional, não-owner
    ThrustForces*                 thrust = nullptr; // opcional, não-owner
    RudderForces*                 rudder = nullptr; // opcional, não-owner
};
std::map<int, BodyEntry> entries_;
```

A API pública muda:
- `state()` → `state(int bodyId)`: query obrigatoriamente por ID.
- `addNavalForceModel(unique_ptr<NavalForceModel>)` → `addNavalForceModel(int bodyId, unique_ptr<NavalForceModel>)`.
- Novo: `registerVessel(int bodyId, DynamicVessel& vessel, ThrustForces*, RudderForces*)`.

O tick interno itera sobre todos os `BodyEntry` na ordem de inserção (preservada pelo
`std::map<int>`). Cada corpo tem seu próprio contexto naval, modelos de força, e
opcionalmente um DynamicVessel.

## Alternativas Consideradas

### Alternativa A: Manter NavalSimulation single-body, criar NavalSimulationMulti separado

- **Descrição**: NavalSimulation preserva API atual (não quebra testes). Uma nova classe
  NavalSimulationMulti suporta N corpos.
- **Prós**: Zero breaking change para testes existentes.
- **Contras**: Duplicação de lógica complexa (buildContext, EMA, tick). Dois objetos com
  semântica quase idêntica é pior do que um breaking change controlado.
- **Por que rejeitada**: O custo de manter duas classes supera o custo de migrar testes.

### Alternativa B: NavalSimulation como wrapper sobre World (antecipando Fase 2)

- **Descrição**: Implementar NavalSimulation como fachada fina sobre World (que terá N
  domínios na Fase 2), preparando a migração.
- **Prós**: Sem retrabalho na Fase 2.
- **Contras**: World não existe ainda; implementar World apenas para satisfazer N-bodies
  na Fase 1 inverte a sequência correta de fases.
- **Por que rejeitada**: Premature. World é Fase 2 explicitamente (D5 em decisions.md).
  NavalSimulation deve ser refatorado na Fase 1 e substituído na Fase 2.

## Consequências

### Positivas

- Simular navio + rebocadores independentes em um único NavalSimulation na Fase 1.
- `buildContext(bodyId)` é chamado separadamente por corpo — sem estado compartilhado entre corpos.
- Preparação limpa para migração para NavalDomain na Fase 2.

### Negativas

- **Breaking change de API**: todos os testes existentes que constroem `NavalSimulation sim(makeTestBody())` precisam ser migrados para:
  ```cpp
  NavalSimulation sim;
  sim.addBody(0, makeTestBody());
  ```
- `state()` sem argumento não existe mais; todos os callers precisam passar o bodyId.

### Riscos

- Risco: tick com múltiplos corpos em `sim_.step(dt)` chama CVODE de cada corpo
  sequencialmente. Com N grande, o custo de tick cresce linearmente. Aceitável para
  Fase 1 (N ≤ 5 corpos navais em cenários típicos).

## Notas de Implementação

- `NavalContext` por corpo: cada `BodyEntry` tem seu próprio `ctx` e `q_avg`. `buildContext(entry, dt)` atualiza apenas o entry passado.
- EMA tau permanece 16.5s por corpo.
- Testes existentes em `tests/simulation/TestNavalInfra.cpp` e `TestNavalIntegration.cpp` são migrados como parte da tarefa de implementação do ADR-004.
