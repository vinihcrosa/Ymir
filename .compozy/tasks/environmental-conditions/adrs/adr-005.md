# ADR-005: C++ EnvironmentTimeline as Keyframe Resolution Layer

## Status

Accepted

## Date

2026-06-29

## Context

The PRD requires temporal keyframe interpolation and vectorial composition of multiple uniform conditions. These computations must resolve to a single `(speed, dirNaut)` pair per condition type before being passed to `Environment` setters. The question is where that resolution logic lives: in TypeScript (frontend resolves, WASM receives per-step values) or in C++ (WASM receives the full timeline once, resolves internally each step).

## Decision

A new C++ class `EnvironmentTimeline` owns the full condition timeline, deserializes it from a JSON string, and resolves to `Environment` setters at each simulation step via `advanceStep(t, env)`. `World::step()` calls `timeline_.advanceStep(time_, env_)` before stepping domains. The frontend serializes the `EnvironmentProfileDTO` to JSON once before starting the simulation and passes it to WASM via a new `loadEnvironment(json)` binding.

## Alternatives Considered

### Alternative A: Frontend-owned resolution

- **Description**: TypeScript store computes interpolated and composed values at every simulation tick, sends resolved `(speed, dir)` to WASM via simple setter bindings each step.
- **Pros**: `Environment.h` and `World.h` unchanged; WASM API minimal (3 setters).
- **Cons**: Frontend must track simulation time independently of the WASM clock. Coupling between JS tick loop and physics time creates drift risk. Phase 3 (real-time instructor changes) makes this fragile — inserting a condition mid-run requires per-step re-evaluation in JS.
- **Why rejected**: Simulation time is authoritative in C++ (`world_->time()`); duplicating it in the JS layer is a source of subtle bugs.

### Alternative B (chosen): C++ EnvironmentTimeline

- **Description**: New `EnvironmentTimeline` class deserializes a JSON payload once, stores keyframe series, and resolves to `Environment` setters each step using the authoritative C++ time.
- **Pros**: Single source of truth for simulation time. Consistent with `dynamics` `Environment.advanceStep()` pattern. Frontend passes data once; no per-step communication for Phase 1 or Phase 2. Phase 3 becomes a targeted mutation of the timeline at a known C++ boundary.
- **Cons**: Requires a JSON parsing library in the C++ core. Two new files in `world` lib.

## Consequences

### Positive

- Resolution logic co-located with physics time — no clock drift.
- Matches the `dynamics` MATLAB reference architecture (`Environment.advanceStep()`).
- Phase 3 (real-time instructor control) becomes a `timeline_.replaceFrom(t, newConditions)` call, not a JS-to-WASM per-step message.
- `NavalDomain::buildContext()` unchanged — still reads `env_` pointer as before.

### Negative

- C++ JSON parsing dependency required (nlohmann/json or equivalent). Must verify it is already available in the project before implementation.
- `World::step()` gains one new call site (`timeline_.advanceStep()`).

### Risks

- JSON schema mismatch between TypeScript serializer and C++ deserializer. Mitigation: shared JSON schema definition in `packages/types`; C++ parser validates required fields and throws on missing keys.

## Implementation Notes

`EnvironmentTimeline` is owned by `World` (same ownership model as `Environment`). `World::step()` calls `timeline_.advanceStep(time_, env_)` before iterating domains. `YmirSimulation::loadEnvironment(json)` passes the string to `world_->timeline().loadJson(json)`. If `timeline_` is empty (no conditions configured), `advanceStep()` is a no-op and `Environment` retains its default zero values.
