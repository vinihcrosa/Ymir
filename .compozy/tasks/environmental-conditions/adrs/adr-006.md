# ADR-006: Worker `loadEnvironment` Message Protocol

## Status

Accepted

## Date

2026-06-29

## Context

`simulation.worker.ts` communicates with `YmirSimulation` (WASM) through a typed message protocol (`start`, `stop`, `loadScenario`, `setActuator`). A new `loadEnvironment(json)` WASM method must be triggered from the frontend. The question is when and how to deliver the environment payload to the worker.

## Decision

A new worker message type `loadEnvironment` is added with a single `json: string` field. `simulationStore.start()` sends `loadEnvironment` immediately after the `ready` handshake and before the `start` message, mirroring how vessel actuator state is re-synced on start. On `reset()`, `loadEnvironment` is re-sent so the timeline is re-applied to the fresh `YmirSimulation` instance. The worker calls `simulation.loadEnvironment(json)` on receipt. If `simulation` is not yet initialized, the message is a no-op (consistent with existing `setActuator` behavior).

## Alternatives Considered

### Alternative 1: Embed environment in `loadScenario` payload

- **Description**: Add `environment` field to the existing `loadScenario` message alongside `vessels`.
- **Pros**: Single message carries full scenario state.
- **Cons**: `loadScenario` reconstructs `YmirSimulation` (WASM object deletion + re-creation). The environment would need to be re-applied after reconstruction, adding ordering complexity. Also, `loadScenario` is sent from `ScenarioStore`; environment is managed by a separate `EnvironmentStore`. Coupling two stores through one message creates an awkward fan-in.
- **Why rejected**: Message coupling and reconstruction ordering are sources of subtle bugs.

### Alternative 2: Per-step `setEnvironment` messages (frontend resolves)

- **Description**: TypeScript sends resolved environment values to the worker each step.
- **Pros**: Trivial worker change.
- **Cons**: Rejected at the architecture level (see ADR-005).
- **Why rejected**: Conflicts with ADR-005 decision.

### Alternative 3: `loadEnvironment` as chosen

- **Description**: Separate message, sent once on start and after reset.
- **Pros**: Decoupled from scenario loading. Clean separation of concerns between `EnvironmentStore` and `ScenarioStore`. Easy to extend to Phase 3 (real-time) by adding a `updateEnvironment` message variant.
- **Cons**: Adds one new message type to the protocol.

## Consequences

### Positive

- `EnvironmentStore` remains decoupled from `ScenarioStore`.
- Pattern is consistent with existing actuator re-sync on start.
- Phase 3 extension is a new message type (`updateEnvironment` for mid-simulation changes), not a modification of this one.

### Negative

- Two-message startup sequence (loadEnvironment → start) requires correct ordering in `simulationStore`.

### Risks

- If `loadEnvironment` arrives after `start`, environment is applied one step late. Mitigation: enforce ordering in `simulationStore.start()` — send `loadEnvironment`, then `start`, using the same synchronous message dispatch order already used for actuator re-sync.

## Implementation Notes

Message shape:
```typescript
{ type: 'loadEnvironment'; json: string }
```

Worker dispatch:
```typescript
case 'loadEnvironment': {
    const json = (e.data as { json: string }).json;
    simulation?.loadEnvironment(json);
    break;
}
```
