# ADR-004: Temporal Keyframe Evolution for All Condition Types

## Status

Accepted

## Date

2026-06-29

## Context

Environmental conditions in real port approach scenarios change over time — tidal currents strengthen and reverse, wind gusts build and subside, swells arrive from different directions as a storm front passes. The MATLAB `dynamics` reference implements temporal interpolation for current via a cell array of time-stamped snapshots (`Environment.advanceStep()`, `currentValue()`). The question is whether to apply this same model to wind and waves.

## Decision

All three condition types — current, wind, and wave — support temporal keyframe evolution using the same model:
- A condition collection is an ordered list of (condition, timestamp) pairs called keyframes.
- At simulation time `t`, the value is the linear interpolation between the two keyframes that bracket `t`.
- A single keyframe (no time evolution) is the degenerate case and must be supported.
- Interpolation for uniform conditions: linear interpolation of direction (angular, wrap-aware) and speed (scalar).
- Interpolation for grid conditions: linear interpolation cell-by-cell of `speedX` and `speedY`.

## Alternatives Considered

### Alternative 1: Temporal Evolution Only for Current

- **Description**: Only current supports keyframes; wind and waves are static per scenario.
- **Pros**: Smaller initial scope.
- **Cons**: Inconsistent model — instructors would need different mental models per type. Wind variation is a common training scenario (storm buildup).
- **Why rejected**: Unified model is simpler to learn and implement once the pattern exists.

### Alternative 2: Continuous Analytical Functions

- **Description**: Instead of keyframes, allow instructors to specify analytical functions (e.g., "wind speed ramps from 5 to 20 m/s over 30 minutes").
- **Pros**: Smooth, physically plausible transitions.
- **Cons**: More complex authoring UI; keyframes already produce smooth interpolated transitions.
- **Why rejected**: Keyframes cover all analytical cases (ramp = two keyframes) and are simpler to author.

## Consequences

### Positive

- Unified mental model: all condition types work the same way.
- Matches the `dynamics` reference implementation for current.
- Enables realistic training scenarios with changing conditions.

### Negative

- Angular interpolation for direction requires care to avoid "wrapping" artifacts (e.g., interpolating between 350° and 10° through 180° instead of through 0°).

### Risks

- Wind direction interpolation must be shortest-path on the circle, not arithmetic average. Must be validated against `dynamics` `changeWind()` convention.
