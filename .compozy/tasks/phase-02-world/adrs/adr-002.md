# ADR-002: Tugs Promovidos de Forças Paramétricas a Corpos Físicos Independentes com CouplingPort

## Status

Accepted

## Date

2026-06-27

## Context

Phase 1 implemented `TugParametricForces` — tugs are static force sources defined by parameters
(push/pull magnitude, direction, application point). They have no body, no heading dynamics,
and no response to the ship's motion. `BerthManeuverSystem` writes tug force demands directly
into `TugParametricForces`.

Phase 2 introduces `CouplingRegistry`, which enables physical bodies to exchange forces. This
creates the opportunity to promote tugs to first-class physics bodies — each a `RigidBody6DOF`
inside `NavalDomain` with its own heading, thrust, and drag.

The question is whether to do this in Phase 2 or defer.

## Decision

Tugs are promoted to independent `NavalDomain` bodies in Phase 2. Each tug is a named entity
with its own `RigidBody6DOF` and a simplified `DynamicVessel` (no waypoint controller needed
for Phase 2 — tugs accept direct force commands).

Force coupling between tug and ship uses `CouplingPort`:
- Tug writes `force` and `interfacePosition` into its provider port each tick.
- Ship reads from the corresponding consumer port; `NavalDomain` injects this force into
  the ship's force accumulator before integration.

`TugParametricForces` is deprecated but kept compilable for one phase. `BerthManeuverSystem`
is updated to write tug force demands into `CouplingPort` rather than `TugParametricForces`.

## Alternatives Considered

### Alternative: Manter TugParametricForces, Usar CouplingPort Apenas para Fase 3

- **Description**: Tugs remain parametric in Phase 2. CouplingRegistry is built but only used for future non-tug coupling.
- **Pros**: Smaller scope for Phase 2; BerthManeuverSystem unchanged.
- **Cons**: Roadmap explicitly states Phase 2 enables tugs as independent bodies. Deferral means Phase 3 must simultaneously deliver API + tug promotion — scope conflict.
- **Why rejected**: The roadmap's Phase 2 entry for CouplingRegistry explicitly mentions "rebocadores como corpos independentes trocando força com o navio".

## Consequences

### Positive

- Tug dynamics (heading lag, thruster response) affect the force delivered to the ship — simulation fidelity increase.
- CouplingPort is validated with a real use case (tug-ship) before Phase 4 uses it for collision response.
- `BerthManeuverSystem` operates on real spatial positions of tugs, not assumed positions.

### Negative

- Tug `RigidBody6DOF` config must be defined (hull parameters, added mass, drag). Needs representative values; exact values are not critical for Phase 2.
- `BerthManeuverSystem` requires a non-trivial update to use CouplingPort.

### Risks

- Risk: Tug physics are underparameterized and produce unrealistic behavior. Mitigation: tugs can use simplified constant-velocity dynamics (drag only, no waves, no squat) with a reduced NavalContext in Phase 2.
