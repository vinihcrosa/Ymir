# ADR-001: Atomic Split Reorganization of Repository Structure

## Status

Accepted

## Date

2026-06-25

## Context

The current repository structure (`core/`, `naval/`, `adapters/`, `applications/`) was created during early prototyping. It does not map to the bounded-context architecture defined in `.prds/prd.md`, which defines five contexts: Physics, World, Vessel, Simulation, and Infrastructure.

The project is still in early stages (no production users, no deployed binaries), making this the lowest-risk moment to restructure. Delaying will compound the cost as more code is added.

The reorganization must:
- Move all existing files to their final bounded context locations under `libs/`
- Rename `applications/` to `apps/`
- Create placeholder directories for future modules (`apps/fast-time/`, `services/`)
- Update all CMakeLists.txt and include paths
- Keep the build green throughout

## Decision

Perform the reorganization as a single atomic commit. All files move to their final destinations in one operation:

- `core/` (RigidBody6DOF, AbstractBody, Forces, integrator/) → `libs/physics/`
- `core/src/Simulation.cpp`, `core/include/ymir/core/Simulation.h` → `libs/simulation/`
- `core/include/ymir/core/math/` → `libs/common/`
- `naval/src/forces/`, `naval/include/ymir/naval/forces/` → `libs/physics/` (force implementations)
- `naval/src/wave/`, `naval/include/ymir/naval/wave/` → `libs/world/`
- `naval/include/ymir/naval/NavalEnvironment.h`, `NavalContext.h` → `libs/world/`
- `naval/include/ymir/naval/VesselConfig.h` → `libs/vessel/`
- `adapters/` → `libs/persistence/`
- `applications/` → `apps/server/`
- CMakeLists.txt files updated to reflect new paths

## Alternatives Considered

### Alternative 1: Rename First, Split Later

- **Description**: First commit renames top-level folders only (core/ → libs/core/, naval/ → libs/naval/). A second commit splits the internals.
- **Pros**: Smaller diffs per commit, easier to bisect if build breaks
- **Cons**: Leaves the structure in an intermediate state that doesn't match the PRD; requires two rounds of CMake edits; more total work
- **Why rejected**: The project is small enough that a single atomic commit is manageable and produces a cleaner git history

### Alternative 2: Create New Structure Without Moving Files

- **Description**: Create the new folder skeleton (libs/, apps/, services/) with CMakeLists.txt and .gitkeep, without moving existing files. Existing code stays in place until reimplemented.
- **Pros**: Zero risk of breaking the build
- **Cons**: Two competing structures coexist indefinitely; confusing for developers and AI agents; old structure would never be cleaned up in practice
- **Why rejected**: Leaving the old structure alongside the new one defeats the purpose of the reorganization

## Consequences

### Positive

- Directory names match PRD bounded contexts 1:1
- Developers and AI agents can navigate the codebase without cross-referencing documentation
- CMake structure is cleaner — one `add_subdirectory` per bounded context
- Foundation is set before significant new code is written

### Negative

- All existing `#include` paths change — every file that includes headers from `core/` or `naval/` must be updated
- `git log --follow` required to trace file history across the move
- CI pipeline must be verified after the commit

### Risks

- **Build breakage**: CMakeLists.txt files must be updated atomically with file moves. Mitigation: run full build and tests locally before committing.
- **Include path updates**: Any missed `#include "ymir/core/..."` becomes a compile error. Mitigation: run `grep -r "ymir/core\|ymir/naval"` after moving to find stragglers.

## Implementation Notes

After moving files, run:
```
grep -r "ymir/core" libs/ tests/ apps/
grep -r "ymir/naval" libs/ tests/ apps/
```
Any hit is an include path that needs updating.
